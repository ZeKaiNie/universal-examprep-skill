# 参与贡献

中文 · [English](CONTRIBUTING.md)

感谢你帮助期末极速备考教练变得更可靠、对学生更有用。我们非常欢迎小而专注的 PR。只要清楚说明用户遇到的问题、不破坏资料来源和学习数据安全，并提供与改动规模相称的验证，大多数 PR 在正常审查和必要调整后都可以合并。即使最初方案还不够完善，维护者通常也会一起协助收敛，而不是简单拒绝。

提交范围清楚的 PR 之前不需要先申请许可。如果准备大幅调整架构，可以先开一个简短 Issue 或 Draft PR，避免重复开发。

## 我们优先看什么

贡献按下面的顺序审查：

1. **调试和可靠性。** 可复现的缺陷、数据丢失或状态损坏、题目与答案遗漏或串题、解析器与校验器契约冲突、来源记录错误、隐私或路径安全问题、跨平台失败和 CI 回归优先级最高。如果能附上一条小而准确的回归测试，会非常有帮助。
2. **真实教材效果和维护。** 可以合法再分发或已妥善匿名化的课程样本、Gold Set、真实课程验收发现、更适合零基础学生的解释、图片和公式质量、语言维护、无障碍体验、性能、包体精简、依赖维护和文档纠错排在第二。请把实际观察到的证据与个人感受分开说明。
3. **新功能。** 一次解决一个明确的学生问题。默认路径应保持轻量；不能破坏学习状态机和来源规则；面向学生的功能应兼顾中英文；重型依赖、联网和上传必须是可选能力，并在用户明确同意后才能启用。
4. **其他有用改进。** 更好的测试和 benchmark 方法、安装与智能体兼容、错误提示、开发工具、翻译、示例和小型清理 PR 也都欢迎。

优先级不是拒绝规则。维护工作尚未结束时，一个范围清楚的新功能仍然可以合并；只是影响正确性和学习数据安全的问题会先处理。

## 值得参考的社区贡献

- [#13——自动生成题目 ID，并规范化判断题答案](https://github.com/ZeKaiNie/universal-examprep-skill/pull/13)，作者 [@ky-2332](https://github.com/ky-2332)。这是很好的调试案例：它复现了真实的 ingestion 到 validation 契约冲突，改动范围集中，根据审查意见继续完善，并补上了回归测试。
- [#1——加入概念疑难追踪](https://github.com/ZeKaiNie/universal-examprep-skill/pull/1)，作者 [@BIueOrange](https://github.com/BIueOrange)。这是聚焦学生问题的好例子：它发现学生除了做错题以外，还需要记录“没有听懂”的概念，而且没有夹带无关功能。后续实现已经继续演进，但问题定义依然很有价值。

这些案例只用于说明方向，不要求每位贡献者照着同一个模板提交。

## 提交 PR 之前

- 说明学生或维护者遇到的问题，以及怎样复现。
- 不要把无关的格式化、重构、生成文件和新功能塞进同一个 PR。
- 除非 PR 明确提供迁移方案，否则应保留现有用户改动和兼容性。
- 不要提交接口密钥、私有课程文件、学生作业、个人信息、生成后的工作区、缓存或无权再分发的试题。
- 如果问题来自真实材料，优先提供可以再分发的最小样本、合成复现，或者只含哈希和元数据的报告；请说明材料许可证或匿名化方式。
- 机器 token、schema、ID、来源凭据和 canonical 状态值保持稳定。翻译面向人的文字，不要翻译机器契约。
- 不要静默增加下载、上传、遥测、远程服务或重型本地依赖。说明边界，并在需要时取得用户明确同意。

## 验证方式

先把功能开发到位，再运行最相关的小范围检查；行为稳定后集中跑完整测试。在 PR 里准确列出运行过什么，以及哪些平台或真实材料检查没有条件执行。

常用命令：

```bash
python -m unittest <relevant.test.module> -v
python -m unittest discover -s tests -v
python scripts/build_dist.py
```

纯文档 PR 不一定需要完整测试。涉及 ingestion、状态、凭据、图片安全、语言契约、打包或 CI 的改动通常需要。不要为了让 CI 变绿而放宽一个本来正确的失败测试；应修复根因，或者清楚说明为什么原契约本身需要调整。

## PR 自查清单

- [ ] PR 说明了问题、范围和用户能看到的结果。
- [ ] 改动集中，没有夹带私有数据或生成文件。
- [ ] 来源、学习状态、题面与答案图片隔离以及隐私边界没有被破坏。
- [ ] 适用时，中英文阅读文档一起更新。
- [ ] 已列出相关测试或真实材料检查，以及已知限制。
- [ ] 新依赖或远程能力保持可选，理由和使用方式已经写清。

如果还不确定 PR 是否已经准备好，可以先开 Draft PR，并说明仍然拿不准的地方。越早诚实地提供上下文，审查越容易；维护者也很乐意一起把有价值的贡献调整到可以合并的状态。
